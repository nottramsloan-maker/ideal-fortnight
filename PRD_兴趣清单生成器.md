## 〇、与当前 Demo 的对齐说明（阅读全文前请先读本节）

本节描述 **仓库当前可运行实现**（以 `apps/web` 路由与页面为准），与后文「目标能力」区分。后文未单独标注 **【Demo】** 的段落，均表示 **产品目标 / 终局设计**，不代表已全部开发完成。

### 〇.1 入口与信息架构（实际）

| 项 | 实际行为 |
|----|----------|
| **默认打开** | 根路径 `/` **重定向到** `/carriers`（「我的载体」——当前稿为房子清单主视觉 + 已完成卡片列表），**不强制**先走 Onboarding。 |
| **Onboarding** | 可通过 **`/onboarding/avatar` → `/onboarding/carrier` → …** 完整走通；其中选「房间」会进 **`/onboarding/first-item`** 拍照后再进 AI 对话；选其他具象载体会进 **`/items/carrier-feed`**（与该载体同构的物品清单流）。 |
| **载体清单目录** | `/carriers/directory`，可从 Onboarding 里「添加载体」进入；支持更换头像子路由。 |
| **添加物品** | `/items/new` 拍照 → **`/items/:id/ai`** AI 对话页 → **`/items/:id/card`** 物品卡片；房子清单上加号会设置「完成后回到清单」等跳转（见 store 逻辑）。 |

### 〇.2 已实现的可交互页面（摘要）

- **Onboarding**：选头像、选载体（见 〇.3）、第一件物品拍照。
- **载体列表**（`/carriers`）：展示已完成条目、打开已生成卡片、加号发起新物品流程、分享卡片区弹层（系统分享 / 下载素材图为 Demo 能力）。
- **载体清单目录**（`/carriers/directory`）、**非房间载体物品流**（`/items/carrier-feed`）。
- **物品拍照**（`/items/new`）、**AI 对话页**（`/items/.../ai`）：固定文案引导（先问名称、再问故事）；支持相册附图、选音频文件（本地 object URL，**无真实语音识别与后端**）；点「完成」后全屏 **thinking** 过渡再进卡片。
- **物品卡片**（`/items/.../card` 与查看态 `item-card-view`）：双屏滑动、纸条与诗句区、点赞、故事编辑、**长按星星预览「他人诗句」** 等 **UI 与本地交互**。

### 〇.3 Demo 中「载体选择」与 PRD 大表的关系

- PRD **§4.2.1 载体类型映射表** 表示 **产品愿景：未来可扩展的载体语义**，**不等于**当前可选列表。
- **当前 Onboarding 载体页**为 **6 个入口**：冰箱、房间、书架、纸箱、显示器、**添加载体**（进入目录流）。**未在别处实现**更多表中类型。

### 〇.4 占位 / 未接设计稿的页面（仅壳或说明文案）

以下路由在代码中为 **开发占位**（`DevPageShell` 或极简占位），**勿在验收 Demo 时按 PRD 全功能理解**：

- `/carriers/:id` 场景编辑、`/carriers/:id/theme` 载体样式、`/carriers/:id/items` 物品列表（独立页）
- `/profile` 个人中心
- `/share/preview/:carrierId` 分享预览
- **外链落地页** `/p/:slug`：仅有 slug 与规则说明占位，**未实现** §4.5.2 完整视觉与列表组件。

### 〇.5 AI、诗句与「社交」在 Demo 中的真实含义

| PRD 表述 | Demo 实际 |
|----------|-----------|
| AI 识别 / 智能填表 / 抠图 | **未接模型与服务**。对话为 **前端流程 + 本地状态**（名称、故事文案、附图、语音文件占位）。 |
| AI 生成诗句 | 卡片诗句来自 **前端预设文案池**（可多套切换），**非**根据用户内容实时调用大模型生成。 |
| 拥有人数、同物件匹配 | 卡片上展示为 **写死的演示数值 / 文案**，**非**平台真实统计。 |
| 别人的 AI 诗句 / 长按看别人故事 | 交互存在时，内容为 **预设文本模拟**，**非**其他用户真实投稿。 |
| 云端存储、多端同步 | **未在本文档 Demo 范围承诺**；当前以浏览器端状态为主（刷新可能丢失，以实现为准）。 |

## 一、产品概述

### 1.1 产品定位

一款面向兴趣收藏爱好者的移动端H5应用,帮助用户快速整理、展示并分享自己的兴趣/收藏清单。通过AI智能识别、场景化载体搭建和仪式感卡片生成,将个人兴趣转化为可传播的社交货币。

### 1.2 核心场景

> "我想把自己在某个兴趣领域的清单整理出来并分享。"

**具体场景举例:**
- 潮玩收藏者展示自己的手办收藏柜
- 图书爱好者分享年度阅读书单
- 游戏玩家整理游戏成就清单
- 旅行者记录旅行纪念品

### 1.3 核心价值主张

| 价值维度 | 描述 |
|----------|------|
| **主体性表达** | 强调"我的"兴趣清单,凸显用户个人品味和收藏故事 |
| **仪式感呈现** | 通过精心设计的载体(书架、展示柜等)搭建场景,增强收藏的珍贵感 |
| **智能效率** | AI识图自动填充物品信息(品牌、年代、稀有度、价格等) |
| **社交传播** | 生成精美的分享卡片,连接同好,发现拥有相同收藏的用户 |

## 二、用户画像

### 2.1 核心用户群体

| 用户类型 | 描述 | 核心诉求 |
|----------|------|----------|
| **深度收藏者** | 拥有大量特定领域收藏品(潮玩、古董、手办等) | 专业展示、记录价值、发现同好 |
| **兴趣爱好者** | 对某领域有浓厚兴趣,收藏数量中等 | 分享爱好、获得认同、拓展社交 |
| **记录型用户** | 偏好整理和记录生活点滴 | 快速记录、美观呈现、便捷回顾 |

### 2.2 用户需求分析

| 需求层级 | 具体需求 | 优先级 |
|----------|----------|--------|
| **核心需求** | 快速记录收藏品、生成精美卡片 | P0 |
| **重要需求** | AI智能识别、搭建场景化载体、社交分享 | P1 |
| **增强需求** | 同好交友、价格评估、稀有度评级 | P2 |

#### 4.1.2 个人主页

**【Demo】** `/profile` 为 **占位页**。与「个人中心」相关的 **可演示能力** 分散在：**`/carriers`（主清单）**、**`/carriers/directory`（载体目录）**、**更换头像子流程**。

| 行李箱 | 纪念品 | 规划中 |

| 衣柜 | 服装 | 规划中 |

#### 4.2.2 载体自定义

**功能描述:** 允许用户对载体进行个性化调整。

**【Demo】** 路由 `/carriers/:id/theme` 等仍为 **占位页**，下列能力 **尚未** 在可运行 Demo 中交付。

**功能要点（目标）:**
- 载体颜色调整
- 背景更换
- 载体尺寸调整
- 装饰元素添加

#### 4.2.3 载体搭建

**功能描述:** 用户将物品放置到载体页面里面,搭建收藏场景。

**【Demo】** 独立「场景编辑」页 `/carriers/:id` 为占位；当前通过 **清单列表 + 物品卡片** 完成「物品进清单」闭环，**非**自由拖拽摆放到场景画布。

**功能要点（目标）:**
- 物品增删

### 4.3 物品模块

#### 4.3.1 拍照/上传

**功能描述:** 用户通过拍照或上传图片的方式添加收藏品

**【Demo】** 已实现拍照/相册选取与进入后续对话链；**裁剪能力**以实现为准（若工程未接则仍为规划项）。

**功能要点（目标）:**
- 支持拍照(调用摄像头)
- 支持相册选择
- 图片裁剪

#### 4.3.2 AI识别

**功能描述（目标）:** 用户先输入物品名字，AI 根据名字来提出相应问题，让用户像给朋友介绍收藏品一样讲述收藏品的故事。

**【Demo 现状】** 对话页为 **固定流程与文案**（先「它叫什么名字？」再「聊聊你和它有什么故事么？」），**后续调用**大模型或后端识别服务；支持用户 **输入文字、选相册图、选音频文件** 写入本地草稿，**现在无/后续有**自动语音转写、**无**自动抠图与字段识别。验收时请勿将本节「AI提问类型」表理解为已全部自动触发。

**AI提问类型（目标产品字段池，供未来接入服务时对齐）:**

| 字段名 | 类型 | 说明 | 是否必问 |
|--------|------|------|----------|
| 物品名称 | 文本 | 物品名称/型号 | 否 |
| 品牌 | 文本 | 品牌/制造商 | 否 |
| 年代 | 文本 | 大约年份/时期 | 否 |
| 稀有度评级 | 评级 | ★☆ 到 ★★★★★ | 否 |
| 价格区间 | 文本 | 市场参考价格 | 否 |
| 城市 | 文本 | 主要市场/来源地 | 否 |
| 拥有人数 | 数字 | 平台内拥有该物品的用户数 | 否 |
| 照片 | 附件 | 物品照片 | 否 |
| 识别时间 | 日期 | AI识别时间 | 否 |

**功能要点（目标）:**
- 自动抠图(去除背景)
- 智能识别物品类型
- 支持手动编辑修正
- 识别语音输入
- 识别文字输入
- 引导用户获得上下文

**【Demo 已具备】** 文字输入、相册附图、语音 **文件** 选择、固定引导文案、完成后进入卡片前 **全屏 thinking 过渡**（时长与动效以实现为准）。

#### 4.3.3 物品卡片生成(含AI诗句)

**功能描述（目标）:** 在用户完成故事与基础信息后,AI结合**全文上下文**(名称、品牌、故事、语音转写、用户上传的照片等)生成**专属诗句**,并合成物品卡片。卡片页提供**同物件社交视角**:展示平台内与该物品(或标准化后的同一物件)关联的用户规模,并支持浏览**其他用户由AI生成的诗句**,形成同好共鸣与创作对比。

**【Demo 现状】** 诗句与部分「他人内容」来自 **前端预设文案与交互模拟**（如诗句多版本切换、长按预览等），**拥有人数等为演示数据**，**不等同**于上线后的真实 AI 生成与平台统计。卡片布局与动效以对应当前 Figma/实现稿为准。

**物品卡片构成元素(版面固定组成,按产品设计排布层级与留白):**

1. **物品抠图** — 去底后的物品主视觉,作为卡片核心图形。  
2. **上传照片(可选)** — 用户补充的照片;无则不展示该区域。  
3. **音频/文字(可选)** — 「我的故事」等:支持纯文字展示,或提供音频播放入口(可与语音转文字稿并列);无音频且无额外文字时可收缩。  
4. **个人头像** — 当前用户的形象/头像,标识该卡片归属。  
5. **AI诗句** — 由AI根据上下文生成的诗句,定稿后固定在卡片中。  
6. **拥有人数** — 平台内与该物品(标准化匹配后的**同一物件**)的持有人数,只读展示。  
7. **别人的AI诗句** — 按住按钮自己诗句的区域变成别人的诗句，松开变回自己的

**物品编辑字段表:**

| 字段名 | 类型 | 说明 | 是否必填 |
|--------|------|------|----------|
| 我的故事 | 文本、语音 | 用户对物品的描述、回忆、故事 | 是 |
| 心情 | Emoji | 当前心情/感受 | 否 | (后期加上)
| 标签 | 文本 | 自定义标签 | 否 | (后期加上)
| 自定义价格 | 数字 | 用户自定义价值 | 否 | (后期加上)
| AI诗句 | 文本 | 由AI根据当前物品上下文生成的诗句,用户可一键重新生成或微调后再定稿 | 否(生成后展示) |
| 拥有人数(同物件) | 数字(只读) | 平台内登记/匹配为**同一物件**的用户数量,对应卡片元素第6项 | 系统展示 |
| 别人的AI诗句 | 列表(只读) | 按住按钮自己诗句的区域变成别人的诗句，松开变回自己的| 系统展示 |

**功能要点（目标）:**
- 支持文字输入、语音转文字;可选音频与卡片元素第3项联动展示
- **上传照片(可选)**:既作为向AI提供上下文,也作为卡片元素第2项在版面中呈现;与物品抠图(第1项)可同时存在
- **AI诗句**:基于当前物品全部已填上下文生成,支持换一版、编辑后定稿,对应卡片元素第5项
- **拥有人数**:对应卡片元素第6项,显著展示(匹配规则:名称标准化、类目、可选条码/型号等,见技术方案)
- **别人的AI诗句**:对应卡片元素第7项,列表或横向滑动浏览;可配合「同好发现」模块(见4.5.3);隐私物品仅展示人数或隐藏他人内容,规则可配置
- **个人头像**:与用户形象设置联动,对应卡片元素第4项

**【Demo 补充】** 已实现卡片 **多屏滑动、故事编辑、点赞、诗句切换、模拟「看别人诗句」类交互** 时，仍以 **本地/预设内容** 为主；与上表「系统展示」「匹配规则」对齐需待后端与算法接入。

#### 4.3.4 物品放置

**功能描述:** 将物品放置到载体上,进行场景搭建。

**【Demo】** 以 **清单内卡片条目** 表达「已收录物品」，**无**画布级拖拽摆放、对齐线与层级编辑。

**功能要点（目标）:**
- 日期显示
- 层级管理
- 对齐辅助线
- 物品增删

### 4.5 社交模块

#### 4.5.1 卡片生成

**功能描述:** 将清单生成分享卡片,支持保存图片及分享至**微信、小红书**等渠道;站外用户通过链接/二维码进入**外链落地页**(非 App 内完整体验),展示规则见 **§4.5.2**。

**【Demo】** 清单页可打开 **分享卡片弹层**，支持 **Web Share API（若环境可用）** 与 **下载分享底图素材**；**不等于**已接微信/小红书 SDK 或生成带短链的正式海报。`/share/preview/...` 为 **占位**。

**卡片元素(成品图/海报层):**
- 用户头像/昵称
- 载体场景图(当前被分享的载体快照)
- 清单/载体名称
- 物品数量
- 清单统计(总价值、稀有度等,可选)
- 二维码或短链入口(指向外链落地页,**不承诺展示完整私密故事**)

**功能要点:**
- 卡片模板
- 卡片效果
- 卡片实时预览
- 高清导出

#### 4.5.2 外站分享展示规则(微信/小红书)

**适用场景:** 用户将**成品分享卡片**或带链接的帖子发到微信会话/朋友圈、小红书笔记等;访客在**微信内置浏览器、小红书内置浏览器或系统浏览器**中打开外链落地页。

**设计原则:** **可看成果与目录,不可深挖他人隐私故事;引导新用户进小程序自建。**

| 规则项 | 要求 |
|--------|------|
| **可见:成品卡片** | 落地页顶部或首屏展示与分享图一致的**视觉成品**(或高清等价渲染),强化品牌与「这是 TA 的兴趣清单」认知。 |
| **可见:对方载体列表** | 展示分享者**载体列表**(名称、封面/缩略图、类型等**目录级信息**),让用户了解对方有哪些兴趣场景。 |
| **不可见/不可点:详细故事** | **不提供**点击进入单件物品的**完整详情页**;不展示「我的故事」全文、私密音频、未公开照片流等。**列表项不可跳转**至故事/物品详情,或点击仅提示「下载/打开小程序查看自己的清单」类弱反馈(具体文案与动效由设计定)。 |
| **转化:进入小程序** | 提供明确主按钮 **「生成我的兴趣清单」**(或等价文案),跳转**微信小程序**(或产品主站小程序),完成授权后进入**创建/初始化流程**,让用户生成**自己的**载体与物品;**不在外链页复刻 App 内完整编辑能力**(避免能力与数据边界混乱)。 |
| **合规与预期** | 外站页需符合各平台对外链/二维码规范;隐私默认**最小披露**,与分享者设置的公开范围一致(若分享者设为仅好友,落地页需对应鉴权或失效提示)。 |

**与设计的协作(Figma):** 需单独输出 **「外链落地页」** Frame 组:含成品卡片区、载体列表区、禁用态/无点击详情的列表样式、底部/浮层 **进小程序** CTA;与 §7.3 统一视觉语言。

**【Demo】** 路由 **`/p/:slug`** 当前仅为 **规则说明占位页**，未实现上表视觉与交互；验收外链规则请以 **本节文字 + 技术方案** 为准，直至设计与前端实现补齐。

#### 4.5.3 同好发现(迭代规划)

**说明:** 站内深度匹配、关注与互动等能力放在版本迭代中展开,详见 §12.2;站内逻辑不与 §4.5.2 外链「不可点故事」冲突。

## 五、用户行为链路

### 5.1 初始化链路(首次启动/首条清单)

**【与 Demo 的关系】** 下图为 **推荐用户动线（目标）**；当前工程 **默认进入 `/carriers`**，用户 **可以不经下图** 直接看到清单。若需演示完整引导，请从 **`/onboarding/avatar`** 进入。

用户首次进入应用时完成形象与首个载体、首件物品的搭建,建立可分享的初始场景。

```
用户设置形象
    ↓
选择载体类型（Demo：房间→拍照首件；其他载体→对应 feed）
    ↓
拍照/上传收藏品
    ↓
「AI」对话录入（Demo：固定问答 + 本地附图/语音文件；无自动抠图与模型识别/要求实现）
    ↓
补充物品信息(我的故事等; Demo 与对话合一/要求实现)
    ↓
thinking 过渡（Demo）→ 物品卡片（Demo：预设诗句与模拟社交数据）
    ↓
回到清单 / 查看已生成卡片（Demo）
    ↓
【目标】放置物品到载体画布、完整场景搭建
    ↓
【目标】生成分享卡片 → 外链落地 → 进小程序
```

### 5.2 初始化后的行为链路(日常使用)

**【Demo】** 当前 **主枢纽为 `/carriers`（房子清单）**，而非已实现的独立「个人中心」页（`/profile` 占位）。下述 **个人中心** 能力在目标与 Demo 之间见 §〇.1、§4.1.2。

完成初始化后,用户以**个人中心**为枢纽,在**载体列表**与**物品列表**之间持续维护内容;分享能力在载体侧触达。

```
【Demo 主路径】我的载体 / 房子清单（/carriers）
    ├── 更换载体目录、添加物品、查看已完成卡片
    └── 分享卡片弹层（素材级）
         ↓
【目标】个人中心（身份、概览）【Demo：占位】
    ↓
载体列表 / 目录【Demo：directory + list】
    ↓
物品录入（拍照 → 对话 → 卡片）【Demo】
【目标】独立「场景编辑 / 载体样式 / 物品列表页」【Demo：占位】
```

**说明（目标）:**
- **个人中心**:身份管理,可随时更换头像/形象,并作为入口进入载体与数据概览。  
- **载体列表**:可新增载体、进入载体场景编辑;在此流程中支持**生成并分享卡片**(清单/场景级分享)。  
- **物品列表**:可添加物品,进入拍照/AI对话/物品卡片等录入与编辑流程(与§5.1中物品相关步骤衔接)。  
- 实际信息架构允许用户在上述模块间**跳转回退**(非严格单线),上图表示主路径优先级；**Demo 已具备的入口以 §〇.1 为准**。

### 5.3 用户诉求分析

| 诉求维度 | 具体需求 | 产品解决方案 |
|----------|----------|--------------|
| 便捷度 | 快速记录,降低输入成本 | AI自动识别,智能填充 |
| 效率 | 很短时间内完成清单搭建 | 批量上传,拖拽放置 |
| 美观度 | 场景搭建漂亮,有仪式感 | 精美载体设计,装饰元素 |
| 社交性 | 方便分享,获得认同 | 一键生卡,同好发现 |

## 六、竞品分析

### 6.1 竞品分类

| 竞品类型 | 代表产品 | 核心优势 | 局限性 |
|----------|----------|----------|--------|

| 差异化维度 | 竞品现状 | 本产品策略 |
|------------|----------|------------|

| **AI能力** | 较少应用AI | AI识图+自动填充,降低门槛 |
| **社交性** | 弱社交或强平台依赖 | 同好发现+轻社交,独立生态 |
| **持续性** | 年度活动型 | 持续使用,日常记录 |
| **情感化** | 数据驱动为主 | "我的故事"+心情记录,情感连接 |

## 七、视觉设计规范

### 7.1 设计风格

- **整体风格:** 手绘、简洁、精致、有温度
- **色彩体系:** 
  - 主色:紫色（#8470FF）
  - 辅色:中性灰系列
- **字体规范（与 Demo 对齐）:**  
  - **中文主字体:** 自定义 **`AaBeiYuLinShiDeXin`**（全站 `--font-family-app`，见工程 `design-tokens`）。  
  - **部分英文/装饰文案:** 如 AI 加载 **thinking……** 使用 **Libre Baskerville Italic**（与 Figma 稿一致）。  
  - **等宽数字等:** 个别列表/开发占位仍可能使用系统 monospace，**不**代表产品终局字体策略。  
- **界面选择:** H5，设计稿常用 **390×844** 逻辑帧（类 iPhone 14 Pro 比例）预览

### 7.2 交互规范

- **操作反馈:** 所有交互操作需有视觉反馈
- **动效设计:** 轻量化、流畅自然
- **手势操作:** 支持拖拽、缩放、旋转

### 7.3 卡片设计

#### 7.3.1 分享成品卡片(导出图)

**卡片层次:**
1. 用户信息区(头像/昵称)
2. 中部:场景展示区(载体+物品)
3. 清单信息区(名称、件数、可选统计)
4. 落地入口区(二维码/品牌标识),扫码行为对应 §4.5.2,**不引导用户预期可在外链阅读他人完整故事**

#### 7.3.2 外链落地页(微信/小红书打开)

与 **§4.5.2** 规则一致,设计需覆盖:
- **上区**:与成品卡视觉连续,突出「他人清单」展示感。  
- **中区**:载体**列表**组件——每项仅目录信息,**无「查看故事」入口**或入口禁用样式明确。  
- **下区/固定底栏**:主 CTA **进入小程序生成自己的清单**;次要可放「了解产品」等。
